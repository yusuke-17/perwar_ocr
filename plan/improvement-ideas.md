## おすすめ優先順位の叩き台

「効きが大きく、コストが小さい」順:

1. C1 検索クエリ正規化 — コスト小で検索の取りこぼしが直る（ほぼバグ修正に近い）
2. B1 変換diff可視化 — 品質チェックの作業が一気に楽になる
3. D1 設定ファイル外部化 — 以降の全機能追加の土台になる
4. A1 画像前処理 — OCR精度の伸びしろが一番大きい本丸
5. D2 進捗バー / D3 ログ — 体感UXと運用性
6. F1 チャット要約 — 価値は最大級。土台（D1）と検索（C1/F2）が整ってからが筋が良い

A2（複数エンジン）・A3（右横書き）は「効きは大きいがコストも大きい本丸」。
土台（D1）と精度の足回り（A1）を整えた後の着手が安全。

## NotebookLM的機能の実現性（案F 詳細）

### 結論
完全に実装可能。しかもこのプロジェクトと相性が良い。
理由は、RAG（検索拡張生成）に必要な部品がすでに揃っているため。

### RAGとは
「質問 → 関連文書を検索 → その本文をLLMに渡して回答させる」仕組み。
LLMの知識だけで答えさせず、手元の資料を根拠に答えさせる。NotebookLMもこの方式。

### 既にある部品（流用できる資産）
- 蓄積データ: library/ の modern.txt（正規化済みテキスト群）
- 検索: SQLite FTS5（質問に関連する文書を引ける）
- ローカルLLM: Ollama + qwen3.5:9b（要約・回答の生成役）
- 制約クリア: すべてローカル完結 → 「外部送信禁止」を満たす（NotebookLMはクラウド送信のためそのままでは使えないが、自前ローカルRAGなら制約に抵触しない）

### 最小構成（まず動かす版）
1. ユーザーが質問を入力
2. 質問を FTS5 で検索し、関連文書の本文を数件取得（C1のクエリ正規化を併用すると精度UP）
3. 取得本文＋質問を qwen3.5 のプロンプトに詰めて回答生成
4. 回答と「使った文書（出典）」を表示

→ 既存資産の組み合わせだけで成立。新規依存ほぼゼロ。

### 発展構成（精度を上げる版）
- F2 セマンティック検索: Ollamaの埋め込みモデル（例 nomic-embed-text / bge-m3）で
  各文書をベクトル化して保存。意味の近さで検索 → 語句が一致しなくても拾える。
  保存先はSQLite（sqlite-vec拡張）かchromadb等。
- F3 出典付き回答: 回答の根拠箇所を文書ID付きで明示 → 誤要約の検証が可能。
- 会話履歴の保持: 直前のやり取りを覚えて追加質問に答える（チャットらしさ）。

### 技術的な注意点
- コンテキスト長: 文書を全部詰めると qwen3.5 の入力上限を超える。
  検索で件数を絞る／長文はチャンク分割（既存 text_modernizer のチャンク機構が流用可能）。
- 要約の正確性: ローカル9Bモデルは大規模クラウドモデルより要約力が劣る。
  出典付き（F3）にして「モデルの作文」と「資料の事実」を区別できるようにすると安全。
- 速度: ローカルLLMは応答に時間がかかる → 進捗表示（D2）があると体感が良い。

### 推奨ステップ
1. 先に C1（クエリ正規化）を入れて検索精度を底上げ
2. F1 最小構成（FTS5検索 + qwen3.5要約）で「動くチャット」を作る
3. 物足りなければ F2（セマンティック検索）→ F3（出典明示）と段階的に強化
